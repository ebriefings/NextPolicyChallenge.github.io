GitHub Tutorial

(for kids that can't GitHub good, and want to do other things good too)

Objective

Most Git and GitHub tutorials youll find online start with the premise that people want to - or more accurately, the presenter thinks you should - understand the principles and mechanics of version control and what Git is. Im going to skip all that and assume that you just want to get working with GitHub with as little difficulty as possible, and that you only want to work with text documents once youre there - not work with computer code. (A deeper question is why GitHub? Im not going to argue that GitHub is the way to do text collaboration, or even a good way. In fact, this tutorial is part of a project that will investigate in part whether GitHub is useful for large scale text collaboration and to try to understand how people use it).

Driving, not engineering

Though you can easily find someone who disagrees with this approach, I believe that - if all you were wanting to do is to learn how to drive - it does not help you right now for me to open up the hood and show you how to check the oil. If, after you have learned how to drive, you want to learn about how a car works, then you will have the context of being able to drive to apply to the mechanisms and principles of how the car works. But our focus today is learning how to get started using GitHub. Once you are collaborating with other people on text documents in GitHub, you can then pull back the layers of the parfait and dig more deeply into working with Git from the command line, understanding the principles of distributed version control and exploring the many facets of GitHub.

Learning Objectives

By the end of this tutorial, you will have:

  1. signed up for a GitHub.com account and set up your account,
  2. created your first project (called a repo or repository) on GitHub.com,
  3. installed and configured Git and GitHub for desktop on you Mac or Windows computer,
  4. cloned a repo from your GitHub.com account to your desktop installation,
  5. made simple changes to a repo locally and synced those changes to your GitHub.com account,
  6. made simple changes to your repo on the GitHub.com site and synced those changes to your local GitHub installation,
  7. found other GitHub users,
  8. forked a repo (I swear, thats what its called - fork a repo - go ahead, say it: its fun) from someone else on GitHub.com,
  9. submitted that changed repo to the original owner of the file as a suggested modification (this is called issue a pull request),
  10. managed pull requests from other people,
  11. updated your fork of the original repo, and
  12. created your own GitHub page.

Some Git and GitHub Vocabulary

This is a small list of what I think are the most important terms to have at the ready. A fuller list can be found here.

  • Branch: a parallel version of repository that does not affect the primary branch allowing you to work freely without disrupting the "live" version. When you've made the changes you want to make, you can merge your branch back into the master branch to publish your changes.
  • Clone: A clone is a copy of a repository that lives on your computer where you can edit the file.
  • Commit: or "revision", is an individual change to a file (or set of files). Every time you save, it creates a unique ID that allows you to keep record of what changes were made, when and by who.
  • Diff: the difference in changes between two commits, visually describing what was added or removed from a file since its last commit.
  • Fork: your copy of another user's repository, where you can freely make changes to a project without affecting the original. Forks remain attached to the original, allowing you to submit a pull request to the original's author to update with your changes. You can also keep your fork up to date by pulling in updates from the original.
  • Git: an open source program for tracking changes in text files.
  • Issue: suggested improvements, tasks or questions related to the repository that can be created by anyone (for public repositories), and are moderated by repository collaborators. Each issue contains its own discussion forum, can be labeled and assigned to a user.
  • Markdown: a simple file format, that makes it easy to write text and have it displayed like a website.
  • Merge: merging takes the changes from one branch and applies them into another. This often happens via a pull request. A merge can be done automatically via a Pull Request via the GitHub.com web interface if there are no conflicting changes, or can always be done via the command line. See Merging a pull request.
  • Pull: getting the latest changes from an online repository and merging them into the file you're both working on (you can fetch without merging, but merging does both).
  • Pull Request: a request to merge proposed changes to a repository submitted by a user and accepted or rejected by a repository's collaborators, with its own discussion forum.
  • Push: sending your committed changes to a remote repository such as GitHub.com after youve changed something locally
  • Repo or repository: a project's folder, containing all of the project files (including documentation) and revision history. Repositories can have multiple collaborators

2. Create your First Repo

On your home page, in the top-right corner, select the + icon and click on New repository to get to this page:

A few things you need to do here before you click Create Repository:

  1. choose a repository name. The easiest thing is to click on the suggested name (petulant-octo-ironman) but feel free to choose your own (letters, numbers and dashes only - no spaces or other punctuation allowed)
  2. write a short description (My first repo! works)
  3. make sure the repo is marked as Public (you have to pay for private repos)
  4. mark the check-box Initialize this repository with a README
  5. Do not add .gitignore.
  6. Think about whether you want to add a licence (but Id suggest leaving it as None; for the Next Policy Challenge, Creative Commons will be recommended - as is used for this tutorial - and that cant be added here).

Congratulations - you just created your first repo!!

3. Install and configured Git and GitHub for desktop on you Mac or Windows computer

Were going to take a real shortcut (like, a wormhole-type shortcut) by installing the native app for Mac or Windows. (If youre using Linux - or if youre really into working from the command-line - youre probably in the wrong tutorial anyway). Because I dont use a Windows computer, Ill have fewer screenshots from that platform, but http://windows.github.com/help.html is really helpful.

Macintosh

If youre using a Mac and running OS 10.7 or higher (GitHub for Mac does not work on OS 10.6 or below; to check what version youre running, click on the in the top-left corner and select About This Mac - the OS X version number will be displayed), go to http://mac.github.com/ and click on Download GitHub for Mac. After you unzip the downloaded file, youll have an application file that you can just drop into your Applications folder. On the webpage youll also see the tab for Help at the top of the page - you may want to bookmark this - http://mac.github.com/help.html - for future reference.

When you launch GitHub for the first time, youll see this setup window. Click Continue to start configuring Git. Include your full name and email address (the one you used to register on GitHub.com) so that Git can correctly attribute all of your work. You'll be given the option to install the "Command Line Utility" for GitHub for Mac. You can skip this if you want - you can always do it later. Add your GitHub.com username (or email address) and password, and click Log in.

Windows

If youre on Windows Vista 7 or 8 (sorry - will not work on Windows XP; but, seriously XP? and what baud modem are you using?), go to http://windows.github.com/ and click on Download GitHub for Windows. Youll also see the tab for Help at the top of the page - you may want to bookmark http://windows.github.com/help.html for future reference.

After you download and install GitHub, youll see this setup window on your first launch. Include your GitHub username or email address (the one you used to register on GitHub.com) so that Git can correctly attribute all of your work, include your password, and click Log In.

Since you made these changes yourself, you dont really need to review them. But notice how the things you deleted are in red, and the stuff you added is in green. Well get better at reading diff files as we use them more.

To commit these changes (i.e., save them locally in GitHub - which is an additional step to the save you just did in TextEdit), type a commit summary (and, optionally, an extended description, if you want to provide more details of the changes you made) in the left-hand pane and click "Commit".

At this point you'll notice that there are now no files selected to commit instead, there are now "Unsynced Commits" at the bottom. This means that your changes have been committed with Git locally but not yet synced to your remote copy on GitHub.com. Before, the icon in the Changes tab was green and red to denote new changes (the same green and red in the diff file) the icon will have now changed to yellow to denote that its been committed but not synced, and you can see that there are Unsynced Commits awaiting your attention.

Take a quick look at your repo on GitHub.com. Notice how it is still the old version, even though youve made a commit on your local version. Youll have to sync both places to update the GitHub.com version.

Sync your changes with GitHub: To send your changed repo up to your copy on GitHub.com, all your have to do is click on the "Sync" button in the Changes view. Once synced, you'll notice that the Changes tab icon is no longer yellow and has changed back to look like the other tabs. This shows that there are no local changes waiting to be committed or synced. Now look at the README file on GitHub.com (refresh the page first) to see that its been changed:

Note: Always commit your changes before you sync. If you dont want to remember that, you can set the GitHub app to Commit & Sync in one step. Click the toggle icon so it turns green and the button says Commit & Sync.

Issue a Pull Request Through GitHub.com (Issue as in transmit)

So maybe you were just being really polite and careful in flagging that issue - but now you definitely have cause to issue a pull request. Merging a pull request

Once youre satisfied with the pull request, you can merge the pull request into your original document. While there are several ways to do this, were only going to cover the very simple way to do it on GitHub.com (because, believe me, it can get complicated). This process assumes there are no conflicts in accepting the pull request and that the pull request can be automatically merged.

While viewing the pull request, in the view, youll see this below all the pull request detail:

Click the "Merge pull request" button, enter a commit message and click "Confirm merge"

This button merges the pull request into the original, sends a notification to the person who initiated the pull request and closes the pull request. Thats one powerful button.

Rejecting a pull request

Maybe youre asking: what if you dont want to accept the proposed changes? Closing a pull request will achieve this, though you will want to let the person who issued the pull request know why.

To do this, on the pull request page, type your reason for rejecting the pull request and then click Close & Comment. Be nice:

11. Updating Your Fork of the Original Repo

Theres one last thing you need to do to keep your GitHub.com version synced with the new updated version of the original repo. Unfortunately, theres no one-button operation for this. Remember what weve done so far:

  • we forked someone elses repo, making a GitHub.com copy and a local GitHub for desktop copy
  • we made some changes (in either place),
  • we issued a pull request, and
  • these were accepted - and the new original repo was changed as a result.

What that now means is that our fork is the {original + our changes}, but the original repo is now in a new version (that has our changes in it). Also, any other changes that may have been made are not reflected in our version.

There are a few steps to get our version synced with the new version of the original. First, in your GitHub desktop app, open the repo youre working with and click on Settings in the left-hand column:

To sync this version with the changed original repo, you need to change the Primary remote repository (origin) to the URL for the original project you forked (i.e., https://github.com/otherpersonsusername/original-repo) and click Update Remote. This is technically called the upstream repository, but that feature is currently not available in the GitHub desktop client.

You then need to synchronize the local repo with the remote repo (you can either use the menu item Repository --> Synchronize, or the key-combination S):

Now your local copy is the same as the version in the original repo. But your version on GitHub.com is different from both of these. To sync that one, you need to again change the Primary remote repository (origin) to the URL for your fork on GitHub.com (i.e., https://github.com/yourusername/repo-name) and click Update Remote. Once more you need to sync the two places using S). Now, the same updated repo exists in three places that you know of - in the original persons GitHub.com account, in your local computer, and in your GitHub.com account.

Rescue: if you ever get really lost about what you have vis-a-vis the original repo, just delete your copy of the repo from your GitHub.com account (this is available through the Settings tab ) and from your GitHub app (using the contextual menu item Remove). Just confirm first that the original is in the safe-keeping of the person you forked it from in the first place. Assuming it is, you can re-fork the original repo to your GitHub.com account and sync it with your desktop app.
